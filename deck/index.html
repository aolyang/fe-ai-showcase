Context Engineering

上下文质量，决定生成质量

该写什么

  • 技术栈与版本（React 18 / TS 5 / pnpm）
  • 硬约束（禁止 class 组件、禁内联 style）
  • 目录与命名约定
  • 测试与提交规范
  • 易错点 / 业务术语
# CLAUDE.md（项目上下文）
- 栈: React 18 + TS 5 + Vite + Tailwind
- 包管理: pnpm（勿用 npm/yarn）
- 组件: 函数组件 + hooks，不放 class
- 样式: Tailwind utility，禁止内联 style
- 测试: Vitest，工具函数都要有用例
- 提交: Conventional Commits

Prompt · 前端场景

五条提示词技巧

  • ① 给约束

    先说栈、版本、风格，再下任务。例：「用 React 18 + TS，不要 any」。

  • ② 给边界

    明确「不要做什么」，比说「做什么」更能收敛结果。

  • ③ 给示例

    贴一个现有组件作为风格参照，比描述一千字管用。

  • ④ 分步，别一次成型

    大模块拆成小步，每步可跑、可验、可回滚。

  • ⑤ 要 diff，不要重写

    让 AI 改动局部并解释 diff，保留你对代码的控制权。

Review & Safety

哪些交给 AI，哪些必须人审

✓适合交给 AI

  • 样板 / 脚手架代码
  • 单元测试与用例
  • 纯函数重构、命名优化
  • 注释、文档、i18n 文案
  • 常规 Bug 的定位与修复建议

✗必须人审

  • 核心业务逻辑与状态机
  • 鉴权、支付、权限边界
  • 敏感数据与密钥处理
  • 架构与依赖决策
  • 性能关键路径与并发

Pitfalls · 反模式

五个常见坑，别踩

盲信生成

能跑 ≠ 正确。不读就提交，迟早出事。

上下文缺失

不给约束就开干，结果风格全乱。

一次写整个模块

大任务不分步，不可验、不可回滚。

跳过测试就合并

AI 写的代码也需要门禁兜底。

把密钥贴进 Prompt

敏感信息一旦出境，无法回收。

不沉淀经验

踩过的坑不写回上下文，下次重来。
